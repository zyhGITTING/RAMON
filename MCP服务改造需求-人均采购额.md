# MCP 服务改造需求 — 人均采购额卡片

## 1. 文档目的

采购部驾驶舱需要计算指定统计期间内的“人均采购额”：

```text
人均采购额 = 四张业务表的采购总额 ÷ 统计期间平均采购部人数
```

驾驶舱会通过 MCP 调用以下 5 个数据源：

1. 人员信息；
2. ERP 采购单明细；
3. 固定资产采购单明细；
4. 托工明细；
5. 其它支出明细。

这些表的数据量较大，**所有明细接口必须先在数据库侧按用户选择的条件过滤，再分页返回**。禁止先全量返回、再由客户端过滤。

---

## 2. 当前接口存在的问题

### 2.1 日期过滤能力不足

现有 MCP 工具只提供 `filters`，其语义是字符串包含匹配，不能可靠表达：

```text
日期 >= start_date AND 日期 <= end_date
```

ERP、固定资产等表的日期字段可能是毫秒时间戳，例如：

```text
1652025600000
```

因此不能使用 `filters: { "os_dd": "2025-01" }` 实现日期筛选，也不能依靠字符串包含匹配筛选年份。

### 2.2 当前 `start_date`、`end_date` 未出现在工具 Schema 中

驾驶舱 `.env` 已准备传递 `start_date` 和 `end_date`，但现有 MCP 工具的 `inputSchema` 没有声明这两个参数，也无法确认服务端是否执行了日期过滤。

### 2.3 其它支出表缺少复核字段

当前其它支出工具定义中未发现“采购数量”和“单价”。如果该表业务上只有费用总额，需要明确：

- 应使用哪个金额字段；
- 金额是含税还是未税；
- 是否允许该表不做“数量 × 单价”复核。

---

## 3. 所有业务明细 MCP 的统一改造要求

ERP、固定资产、托工、其它支出四个业务明细工具应统一支持以下请求参数。

### 3.1 推荐的 Input Schema

```json
{
  "type": "object",
  "properties": {
    "start_date": {
      "type": "string",
      "format": "date",
      "description": "统计开始日期，格式 YYYY-MM-DD，包含当天"
    },
    "end_date": {
      "type": "string",
      "format": "date",
      "description": "统计结束日期，格式 YYYY-MM-DD，包含当天"
    },
    "page": {
      "type": "integer",
      "minimum": 1,
      "default": 1
    },
    "page_size": {
      "type": "integer",
      "minimum": 1,
      "maximum": 1000,
      "default": 200
    },
    "filters": {
      "type": "object",
      "description": "其它字段级过滤条件",
      "additionalProperties": {
        "type": "string"
      }
    }
  },
  "required": ["start_date", "end_date"]
}
```

### 3.2 参数校验

服务端必须校验：

- 日期格式必须是 `YYYY-MM-DD`；
- `start_date <= end_date`；
- 两端日期均为闭区间；
- `page >= 1`；
- `page_size` 不超过服务端上限；
- 未传日期范围时拒绝业务明细查询，不允许默认全量查询；
- 建议限制单次统计跨度，例如不超过 366 天。超过时返回明确错误，避免误操作。

错误示例：

```json
{
  "code": 400,
  "message": "start_date 和 end_date 必填，格式为 YYYY-MM-DD，且查询跨度不能超过 366 天"
}
```

### 3.3 数据库侧日期过滤

日期过滤必须在分页之前执行：

```sql
SELECT ...
FROM source_table
WHERE business_date >= :start_date
  AND business_date < (:end_date::date + INTERVAL '1 day')
ORDER BY business_date, stable_id
LIMIT :page_size
OFFSET (:page - 1) * :page_size;
```

如果原字段是 13 位 Unix 毫秒时间戳，可参考：

```sql
WHERE to_timestamp(os_dd::bigint / 1000)::date
      BETWEEN :start_date::date AND :end_date::date
```

建议优先使用不破坏索引的写法：在数据库中增加可索引的日期生成列，或先将查询参数转换成毫秒时间戳再比较原字段。

```sql
WHERE os_dd::bigint >= :start_timestamp_ms
  AND os_dd::bigint <  :end_exclusive_timestamp_ms
```

其中：

- `start_timestamp_ms` 为开始日 00:00:00 的毫秒时间戳；
- `end_exclusive_timestamp_ms` 为结束日下一天 00:00:00 的毫秒时间戳。

### 3.4 分页必须在过滤之后执行

正确顺序：

```text
日期范围过滤 → 其它条件过滤 → 稳定排序 → 分页 → 返回
```

禁止：

```text
全表分页 → 取第一页 → 客户端本地判断日期
```

建议响应中包含：

```json
{
  "data": [],
  "page": 1,
  "page_size": 200,
  "total": 1250,
  "has_next": true
}
```

如果不方便计算 `total`，至少需要准确返回 `has_next`。

### 3.5 稳定分页

必须使用稳定且唯一的排序条件，例如：

```sql
ORDER BY business_date ASC, document_no ASC, line_no ASC
```

不能只按日期排序，否则同日期大量记录可能在翻页过程中重复或遗漏。

## 4. 人员 MCP 改造要求

当前已确认的人员字段：

| 业务含义 | 接口字段 |
|---|---|
| 工号 | `yuangong_bh` |
| 部门 | `fbm` |
| 入职日期 | `frzrq` |
| 姓名 | `fxm` |
| 员工状态 | `fzt` |
| 离职日期 | `fzxrq` |

采购部精确部门值为：

```text
/衡阳镭目/采购部
```

### 4.1 请求示例

```json
{
  "filters": {
    "fbm": "/衡阳镭目/采购部"
  },
  "page": 1,
  "page_size": 2000
}
```

人员数据量较小，但仍必须先在数据库侧按部门过滤。

### 4.2 人员返回要求

- 必须包含在职人员和已离职人员；
- 离职人员必须返回离职日期；
- 工号必须以字符串返回，避免前导零或格式变化；
- 空离职日期统一返回空字符串或 `null`；
- 历史人员不得因为当前状态为离职而被直接排除。

驾驶舱会根据工号第 4–7 位推导入职年月，并结合离职日期计算历史月份人数。

## 5. 各业务表需要返回的字段

四张业务表应尽量统一返回以下标准含义。原字段名可以不同，但工具描述中必须明确对应关系。

| 标准含义 | 是否必需 | 说明 |
|---|---:|---|
| 业务日期 | 是 | 用于统计期间过滤，统一返回 `YYYY-MM-DD` 最佳 |
| 业务员 | 是 | 后续支持按采购员钻取 |
| 采购数量 | 原则上是 | 数值类型或可转换的数值字符串 |
| 单价 | 原则上是 | 必须明确含税或未税 |
| 采购总价 | 建议 | 有则直接计入，并用数量 × 单价复核 |
| 单据号 | 是 | 用于异常定位和对账 |
| 行号/唯一 ID | 是 | 用于稳定分页、去重和异常定位 |

### 5.1 ERP 采购单明细

当前样本已确认：

| 业务含义 | 字段 | 备注 |
|---|---|---|
| 日期 | `os_dd` | 当前为 13 位毫秒时间戳 |
| 业务员 | `sal_name` | |
| 数量 | `qty` | |
| 单价 | `up` | 样本中与 `qty × up = amt` 一致 |
| 含税总价 | `amt` | 推荐作为当前汇总金额候选 |
| 未税总价 | `amtn` | 与 `amt` 的税口径不同 |
| 单据号 | `os_no` | |
| 行号 | `itm` 或稳定唯一键 | 需确认唯一性 |

需要 MCP 开发者确认 `up` 和 `amt` 是否始终为同一税口径。目前样本表现为：

```text
qty × up = amt
```

### 5.2 固定资产采购单明细

工具定义显示可能使用：

| 业务含义 | 候选字段 |
|---|---|
| 日期 | `os_dd` |
| 数量 | `qty` |
| 单价 | `up` |
| 总价 | `amt` 或 `amtn_net` |
| 单据号 | `os_no` |
| 业务员 | `sal_no_pona`、`sal_no_usena` 或其它字段，需明确 |

请在工具描述中明确业务员字段、含税金额字段和未税金额字段。

### 5.3 托工明细

当前约定字段：

| 业务含义 | 字段 | 备注 |
|---|---|---|
| 日期 | `TW_DD` | `YYYY-MM-DD` |
| 制单人 | `USR_NAME` | 暂代业务员，待业务确认 |
| 数量 | `QTY_MRP` | 制造数量 |
| 单价 | `UP` | 外协单价 |
| 总价 | `AMTN` | 加工未税金额 |
| 单据号 | `TW_NO` | |

托工接口也需要在服务端按 `TW_DD` 执行起止日期过滤。

### 5.4 其它支出明细

当前工具定义中存在：

```text
ep_dd、ep_no、usr_name、amt、amtn、amtn_net
```

但未发现采购数量和单价。请明确：

1. 业务日期是否为 `ep_dd`；
2. 业务员是否为 `usr_name`；
3. 含税金额使用哪个字段；
4. 未税金额使用哪个字段；
5. 其它支出是否天然只有费用总额，没有数量和单价；
6. 若没有数量和单价，是否允许直接按总额计入，并豁免乘法复核。

## 6. 金额和数据类型要求

### 6.1 税口径必须明确

四张表最终相加前必须采用同一口径：

- 全部含税；或
- 全部未税。

每个金额和单价字段必须在工具描述中标明“含税/未税”和币种。

### 6.2 数值字段

数量、单价、金额可以返回 JSON 数值，也可以返回可转换的数值字符串，但不能包含：

- 千位分隔符；
- 中文单位；
- 货币符号；
- 无法解析的占位字符。

空值统一返回 `null`，不要用 `-`、`无` 等文本。

### 6.3 日期字段

建议 MCP 在返回层统一把业务日期转换为：

```text
YYYY-MM-DD
```

如果继续返回毫秒时间戳，必须在工具描述中明确：

- 10 位秒时间戳还是 13 位毫秒时间戳；
- 使用的时区；
- 日期边界如何处理。

本项目按 `Asia/Shanghai` 业务日期理解。

如果允许 MCP 直接返回标准字段，推荐：

```json
{
  "data": [

      "row_id": "PO25010001-1",
      "document_no": "PO25010001",

  "page": 1,
  "page_size": 200,
  "total": 1,
  "has_next": false
}
```

如果必须返回原字段，也可以，但必须保证工具描述中有完整字段说明。

## 8. 性能与安全要求

- 日期字段、部门字段及稳定排序键应建立索引；
- 必须先过滤再分页；
- 日志只能记录工具名、期间、页码、行数和耗时；
- 日志不得记录 Authorization、完整人员信息或采购明细正文；
- 建议设置查询超时和最大返回页数；
- 同一统计期间的重复请求可以在服务端短期缓存；
- MCP 鉴权失败、参数错误和数据库超时必须返回不同的明确错误信息。

## 9. 验收测试

### 9.1 人员过滤

请求：

```json
{
  "filters": {
    "fbm": "/衡阳镭目/采购部"
  },
  "page": 1,
  "page_size": 2000
}
```

验收：

- 返回记录全部属于采购部；
- 包含在职和离职人员；
- 包含工号、入职日期和离职日期；
- 不返回其它部门人员。

### 9.2 单日边界

```json
{
  "start_date": "2025-01-01",
  "end_date": "2025-01-01",
  "page": 1,
  "page_size": 200
}
```

验收：仅返回 2025-01-01 当天业务记录。

### 9.3 月度查询

```json
{
  "start_date": "2025-05-01",
  "end_date": "2025-05-31",
  "page": 1,
  "page_size": 200
}
```

验收：所有返回记录日期均落在 2025 年 5 月。

### 9.4 年度查询

```json
{
  "start_date": "2025-01-01",
  "end_date": "2025-12-31",
  "page": 1,
  "page_size": 200
}
```

验收：

- 数据库执行计划中存在日期范围条件；
- 不发生全表数据传输；
- 翻页后没有重复或遗漏；
- 所有日期均在闭区间内。

### 9.5 非法请求

以下请求应直接拒绝：

- 不传 `start_date/end_date`；
- `start_date > end_date`；
- 日期格式错误；
- 查询跨度超过服务端限制；
- `page_size` 超过最大值。

### 9.6 金额复核样本

每张表提供至少 5 条可人工核对的样本，验证：

```text
数量 × 单价 ≈ 总价
```

允许误差暂定：

```text
|数量 × 单价 - 总价| <= max(0.01, |总价| × 0.5%)
```

如果某表没有数量或单价，必须在接口文档中明确该表豁免复核，并说明总额字段口径。

## 10. 改造完成后的交付内容

请提供：

1. 5 个 MCP 工具最新的 `tools/list` 输出；
2. 每个工具的完整 Input Schema；
3. 各字段业务含义、税口径、币种和日期格式；
4. 每个工具一组带日期范围的请求/响应样例；
5. 分页和 `has_next/total` 的定义；
6. 日期范围过滤确实在数据库侧执行的确认；
7. 其它支出表没有数量/单价时的业务处理结论。

完成以上改造后，驾驶舱桥接层即可安全地按用户选择的日历年、日历月或自定义区间调用 MCP，并计算人均采购额卡片。
